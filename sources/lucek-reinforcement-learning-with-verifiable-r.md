# Reinforcement Learning with Verifiable Rewards (RLVR) for LLMs

### Core Problem
While Supervised Fine-Tuning (SFT) and preference-based Reinforcement Learning from Human Feedback (RLHF) are standard for LLM alignment, they face significant bottlenecks. SFT is limited by the saturation of high-quality pre-training data and diminishing returns in scaling. RLHF relies on subjective human or AI preferences, which can be inconsistent and difficult to scale. There is a need for a training methodology that optimizes for objective, global outcomes and procedural generalization rather than subjective desirability.

### Method and Recipe
Reinforcement Learning with Verifiable Rewards (RLVR) shifts the reward signal from subjective preference to automatic, objective verification (e.g., correctness). The process follows these steps:

1.  **Establish a Reference Policy:** A base LLM undergoes SFT to ensure basic instruction following, formatting consistency, and domain knowledge.
2.  **Define the RL Environment:** An environment is constructed consisting of:
    *   **Action Space:** The tokens or sequences generated by the LLM.
    *   **Task Context:** The prompts and constraints the model must navigate.
    *   **Verifiability:** The environment must provide clear, ideally binary, signals (Pass/Fail) to ensure learning efficiency and eliminate ambiguity.
3.  **Implement Reward Functions:** Rewards are categorized into three tiers:
    *   **Accuracy Rewards:** Binary verification of objective correctness (e.g., unit tests for code, ground-truth matching for math).
    *   **Format and Structural Rewards:** Constraints on output organization (e.g., requiring reasoning within `<think>` tags or enforcing JSON schemas).
    *   **Heuristic Proxies/Generative Evals:** Used when full verification is impossible. These include tool-usage efficiency, partial matching, or "LLM-as-a-judge" for nuanced feedback.
4.  **Policy Optimization:** The LLM (the agent) interacts with the environment, and RL algorithms (such as Proximal Policy Optimization) adjust the model weights to maximize the cumulative reward.

**Example Implementation (PII Masking):**
To teach an LLM to redact personally identifiable information (PII), the author utilized the `Verifiers` library with the following configuration:
*   **Dataset:** A 30k subset of US English examples from `open-pii-masking-500k-ai4privacy`.
*   **Reward Rubric:**
    *   **Accuracy:** Binary check for exact match with expected masked output (Weight: $1.0$).
    *   **PII Count:** Binary check if the number of `[PII]` tags matches the expected count (Weight: $0.5$).
    *   **Format:** Binary check for the presence of `<masked_output>` XML tags (Weight: $0.1$).

### Key Quantitative Results and Emergent Behaviors
The source highlights that RLVR enables "emergent reasoning," where models learn to construct and adapt procedural sequences to achieve verifiable outcomes.

*   **General Success:** Models like OpenAI’s o series and DeepSeek-R1 demonstrated that verifiable rewards incentivize the development of reasoning capabilities.
*   **Procedural Improvements:** Research using the `AppWorld` environment (simulating consumer apps) showed that RL training led LLMs to adopt sophisticated behaviors despite only receiving a simple task completion reward:
    *   Transitioning from open-loop control to checking tool results.
    *   Consistently reading API documentation before making calls.
    *   Reducing assumptions about system states and placeholder values.
    *   Persisting through API errors.

### Limitations
The primary limitation of RLVR is the dependency on verifiability. The "Verifier's Rule" states that the ease of training is proportional to how verifiable the task is. Furthermore, while heuristic proxies and LLM-based rewards can be used for non-binary tasks, they are prone to introducing unintended bias and instability, which can undermine the objective benefits of the RLVR approach.
