# Learning to Generate Unit Test via Adversarial Reinforcement Learning (UTRL)

### Core Problem
Generating high-quality unit tests is labor-intensive and requires deep reasoning to cover challenging edge cases. While Large Language Models (LLMs) can automate this, existing supervised fine-tuning (SFT) methods depend on costly ground-truth unit test labels, which are difficult to scale across diverse programming domains. The central challenge is designing a reward function that can reliably assess the quality of generated unit tests without relying on ground-truth annotations.

### Method: UTRL Framework
UTRL is an adversarial reinforcement learning (RL) framework that iteratively trains a unit test generator ($\mathcal{M}_{\text{UT}}$) and a code generator ($\mathcal{M}_{\text{code}}$). The framework utilizes a dataset of instruction-code solution pairs $\mathcal{D} = \{(I_k, C_k^*)\}$, removing the need for ground-truth unit test labels.

**The iterative recipe is as follows:**
1.  **Train Unit Test Generator ($\mathcal{M}_{\text{UT}}$):** For a given instruction $I$, $\mathcal{M}_{\text{UT}}$ generates a unit test $\mathcal{T}$. This model is optimized to maximize a combined reward $r_{\text{UT}}$ consisting of:
    *   **Discrimination Reward ($R_{\text{disc}}$):** Rewards $\mathcal{M}_{\text{UT}}$ when its tests successfully distinguish the ground-truth code $C^*$ from imperfect code $\mathcal{C}$ generated by $\mathcal{M}_{\text{code}}$.
    *   **Validity Reward ($R_{\text{valid}}$):** Ensures the generated test cases are functionally valid by verifying they pass under the ground-truth code $C^*$.
2.  **Train Code Generator ($\mathcal{M}_{\text{code}}$):** $\mathcal{M}_{\text{code}}$ is trained to produce code $C$ that maximizes the pass rate over the unit tests $\mathcal{T}$ generated by the current $\mathcal{M}_{\text{UT}}$.
3.  **Iteration:** As $\mathcal{M}_{\text{code}}$ learns to produce more correct solutions, $\mathcal{M}_{\text{UT}}$ must learn to generate increasingly discriminative test cases to detect subtler faults, creating an adversarial co-evolution.

### Key Formulas
The **Discrimination Reward** is defined as the fraction of sampled code solutions $\mathcal{C}$ detected by at least one valid test case:

$$
R_{\text{disc}}(\mathcal{T}, \mathcal{C}, C^*) = \frac{1}{|\mathcal{C}|} \sum_{C \in \mathcal{C}} \left[ 1 - \prod_{T \in \mathcal{T}} (1 - \text{Pass}(C, T))^{\text{Pass}(C^*, T)} \right]
$$

The **Validity Reward** uses a hyperparameter $\tau$ to prevent the model from maximizing the reward by generating only a few trivial test cases:

$$
R_{\text{valid}}(\mathcal{T}, C^*, \tau) = \frac{\sum_{T \in \mathcal{T}} \text{Pass}(C^*, T)}{\max(|\mathcal{T}|, \tau)}
$$

The **Total Unit Test Reward** is a weighted sum:

$$
r_{\text{UT}} = \lambda R_{\text{disc}}(\mathcal{T}, \mathcal{C}, C^*) + (1 - \lambda) R_{\text{valid}}(\mathcal{T}, C^*, \tau)
$$

The **Code Reward** for $\mathcal{M}_{\text{code}}$ is the proportion of functionally valid test cases passed by the generated code:

$$
R_{\mathrm{code}}(C, \mathcal{T}, C^*) = \frac{\sum_{T \in \mathcal{T}} \text{Pass}(C, T) \cdot \text{Pass}(C^*, T)}{\sum_{T \in \mathcal{T}} \text{Pass}(C^*, T)}
$$

### Key Quantitative Results
Experiments using Qwen3-4B and Qwen3-14B on the TACO and LiveCodeBench datasets demonstrate the following:
*   **Best-of-N Improvement:** When used as evaluators for best-of-32 sampling with a Qwen3-8B generator, unit tests from UTRL-trained Qwen3-4B achieved **14.9% code accuracy**, significantly higher than SFT-trained Qwen3-4B (**11.7%**) and frontier models like GPT-4.1 (**13.3%**).
*   **Unit Test Fidelity:** UTRL-trained Qwen3-14B achieved a Spearman's correlation of **0.827**, outperforming GPT-4.1 in replicating the evaluation induced by ground-truth tests.
*   **Code Generation:** The code generator trained via UTRL achieved a **15.3% pass@1 accuracy**, which is comparable to the upper bound of 15.9% achieved by training on ground-truth unit tests.
*   **Comparison to CURE:** UTRL outperformed CURE (an RL baseline requiring unit test pairs), achieving code accuracies of **14.1% vs 9.7%** (for Qwen3-8B generator) and higher unit test fidelity (**0.593 vs CURE**).

### Stated Limitations
1.  **Performance Gap:** A gap still exists between UTRL-generated tests and rigorously verified ground-truth unit tests.
2.  **Domain Scope:** Experiments were limited to competitive programming; the framework's effectiveness in broader software engineering domains remains to be scaled.
3.  **Fixed Length:** The current implementation trains the model to generate a fixed number of test cases rather than adaptively determining the required number based on task complexity.
