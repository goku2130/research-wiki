# Planning-Guided Transformer Decoding (PG-TD) for Code Generation

### Core Problem
Conventional Transformer decoding algorithms for code generation, such as beam search and sampling, optimize for the likelihood of token sequences rather than the functional correctness of the resulting program. While "sampling + filtering" pipelines (e.g., AlphaCode) use test cases to filter candidates after generation, they do not incorporate test-case feedback *during* the decoding process. This leads to sample inefficiency, as a single incorrect token can render a program entirely non-functional, requiring exhaustive sampling to find a correct solution.

### Method: Planning-Guided Transformer Decoding (PG-TD)
PG-TD treats code generation as a Markov Decision Process (MDP) and integrates a planning algorithm inspired by Monte-Carlo Tree Search (MCTS) to perform lookahead search guided by program pass rates.

**Step-by-Step Recipe:**
1.  **MDP Formulation**: 
    *   **State ($s$)**: The concatenation of the problem description and a partial or complete program.
    *   **Action ($a$)**: A token from the Transformer's vocabulary.
    *   **Reward**: The pass rate of a complete program on public test cases; partial programs receive a reward of 0.
2.  **Selection**: The algorithm starts at the root and recursively selects branches using a Predictor Upper Confidence Bound (P-UCB) heuristic, which balances exploitation of high-value states, exploration of under-visited states, and the Transformer's prior probabilities.
3.  **Expansion**: The Transformer identifies the top-$k$ most likely next tokens to create new child nodes.
4.  **Evaluation**: To estimate the value of a partial program (node), the algorithm calls a `BEAM SEARCH` function (with beam size $b$) to complete the program. The resulting complete program is executed against public test cases to determine the reward.
5.  **Backpropagation**: The reward is propagated back up the tree to update the values ($Q$) of the node and its ancestors.
6.  **Efficiency Mechanisms**:
    *   **Tree Structure Caching**: Reuses the implicit tree generated during the Transformer's beam search to avoid redundant computations.
    *   **Sequence Caching**: Caches complete programs and their rewards to avoid re-running beam search for previously encountered prefixes.

### Key Formulas
The selection process is governed by the **P-UCB** formula:

$$
\mathrm{P-UCB}(s,a)=Q(s,a)+\beta(s)\cdot P_{\mathrm{Transformer}}(a|s)\cdot\frac{\sqrt{\log(s.visits)}}{1+s'.visits}
$$

Where the exploration weight $\beta(s)$ is defined as:

$$
\beta(s)=\log\left(\frac{s.visits+c_{base}+1}{c_{base}}\right)+c
$$

For controllable code generation, the reward function $\mathcal{R}$ is modified:
*   **Code Length Penalty**: $\mathcal{R}_{\mathrm{length}}=p+\lambda_{1}\times e^{-l_{c}/t}$
*   **Comment Encouragement**: $\mathcal{R}_{\mathrm{comment}}=p+\lambda_{1}\times e^{-l_{c}/t}+\lambda_{2}\times\min(1,\frac{N_{cm}}{N_{max}})$
*(Where $p$ is the pass rate, $l_c$ is code length, and $N_{cm}$ is the number of comment lines).*

### Key Quantitative Results
PG-TD was evaluated using GPT-2 and GPT-Neo backbones on the APPS and CodeContests benchmarks.

*   **Pass Rate Improvements (APPS Introductory)**: Using APPS GPT-2, PG-TD ($c=4$) achieved a pass rate of **26.70%**, outperforming Sampling+Filtering (**25.19%**) and Beam Search (**11.95%**). With GPT-Neo, PG-TD achieved **29.27%** vs. Sampling+Filtering's **27.71%**.
*   **CodeContests**: PG-TD achieved a pass rate of **24.05%** compared to **20.40%** for Sampling+Filtering.
*   **Computational Efficiency**: On 100 APPS introductory problems, implementing both caching methods reduced computation time from **2206.38s** (no caching) to **1312.27s**.
*   **Error Reduction**: PG-TD significantly reduced failure rates compared to Beam Search, lowering compilation errors from **5.58% to 1.93%** and runtime errors from **32.95% to 19.5%**.
*   **Finetuning**: Finetuning a GPT-2 (2.7B) model using high-quality samples generated by PG-TD improved the pass rate from **14.32%** (original) to **16.54%**.

### Stated Limitations
1.  **Reliance on Test Cases**: The algorithm requires test cases to compute rewards, although the authors note that automatically generated test cases can mitigate this.
2.  **Computational Cost**: PG-TD is more expensive than pure beam search because it requires multiple beam search calls within the tree search loop.
